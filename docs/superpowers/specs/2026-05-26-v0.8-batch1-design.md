# v0.8 Batch 1 设计：部署与安全基础

**日期：** 2026-05-26
**版本：** v0.8 Batch 1
**前置版本：** v0.7（已完成）

---

## 概述

Batch 1 聚焦"部署与安全基础"，三个功能相互独立、无前置依赖：

1. **Docker 部署** — 一键容器化启动
2. **TLS/SSL** — WebSocket 加密传输（wss://）
3. **限流中间件** — 连接限流 + 消息频率限制

产出：项目可一键 Docker 部署，支持 wss 加密连接，有基础防护。

## 1. Docker 部署

### 1.1 目标

- 多阶段构建 Dockerfile，最终镜像只包含 Go binary + Alpine 基础
- docker-compose.yml 一键启动 server + SQLite volume 持久化
- 预留 Redis/NATS 服务定义（注释），为 Batch 3 做准备

### 1.2 Dockerfile

```dockerfile
# Stage 1: Build
FROM golang:1.26-alpine AS builder
RUN apk add --no-cache gcc musl-dev
WORKDIR /build
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=1 GOOS=linux go build -o qqgo-server ./cmd/server

# Stage 2: Runtime
FROM alpine:3.21
RUN apk add --no-cache ca-certificates
WORKDIR /app
COPY --from=builder /build/qqgo-server .
EXPOSE 8080
ENV DB_PATH=/data/qqgo.db
VOLUME ["/data"]
CMD ["./qqgo-server"]
```

**关键决策：**
- SQLite 驱动 `modernc.org/sqlite` 是纯 Go 实现，但构建时仍需要 CGO（因为底层依赖 cgo 编译）
- `ca-certificates` 用于 TLS 根证书验证
- 数据目录统一 `/data`，通过 `DB_PATH` 环境变量指向

## 2. TLS/SSL

### 2.1 目标

- 服务端支持 cert/key 路径配置，有证书时启动 wss://，无时降级 ws://
- 提供自签证书生成脚本
- 客户端支持 wss:// 连接

### 2.2 服务端配置变更

**`internal/config/config.go` 新增字段：**

```go
type ServerConfig struct {
    // ... existing fields ...
    TLSCert string // env: TLS_CERT, default: ""
    TLSKey  string // env: TLS_KEY, default: ""
}
```

**环境变量：**

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `TLS_CERT` | `""` | TLS 证书文件路径 |
| `TLS_KEY` | `""` | TLS 私钥文件路径 |

### 2.3 服务端启动逻辑变更

**`cmd/server/main.go`：**

```go
// 当前：
srv.ListenAndServe()

// 改为：
if cfg.TLSCert != "" && cfg.TLSKey != "" {
    log.Printf("starting wss://%s:%s", cfg.Host, cfg.Port)
    srv.ListenAndServeTLS(cfg.TLSCert, cfg.TLSKey)
} else {
    log.Printf("starting ws://%s:%s", cfg.Host, cfg.Port)
    srv.ListenAndServe()
}
```

### 2.4 自签证书生成脚本

**`scripts/gen-cert.sh`：**

```bash
#!/bin/bash
set -e
CERT_DIR="certs"
mkdir -p "$CERT_DIR"
openssl req -x509 -newkey rsa:4096 \
  -keyout "$CERT_DIR/server.key" \
  -out "$CERT_DIR/server.crt" \
  -days 365 -nodes \
  -subj "/CN=localhost/O=QQGO/C=CN"
echo "Generated: $CERT_DIR/server.crt, $CERT_DIR/server.key"
```

### 2.5 客户端变更

**连接 URL 改为可配置：**

```go
// 当前：
url := "ws://localhost:8080/ws"

// 改为：
scheme := os.Getenv("SERVER_SCHEME") // default "ws"
if scheme == "" { scheme = "ws" }
host := os.Getenv("SERVER_HOST")     // default "localhost"
if host == "" { host = "localhost" }
port := os.Getenv("SERVER_PORT")     // default "8080"
if port == "" { port = "8080" }
url := fmt.Sprintf("%s://%s:%s/ws", scheme, host, port)
```

**wss 自签证书支持：**

```go
dialer := websocket.DefaultDialer
if scheme == "wss" {
    dialer.TLSClientConfig = &tls.Config{InsecureSkipVerify: true}
}
conn, _, err := dialer.Dial(url, nil)
```

**新增环境变量：**

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `SERVER_SCHEME` | `ws` | 连接协议：`ws` 或 `wss` |

### 2.6 验证方式

```bash
# 生成自签证书
bash scripts/gen-cert.sh
# 启动服务端（TLS）
TLS_CERT=certs/server.crt TLS_KEY=certs/server.key go run ./cmd/server
# 客户端连接（wss）
SERVER_SCHEME=wss go run ./cmd/client
# 验证：握手成功，消息正常收发
```

## 3. 限流中间件

### 3.1 目标

- HTTP 层连接限流：最大并发 WebSocket 连接数（环境变量 MAX_CONNECTIONS）
- 消息频率限制：每用户每秒最多 N 条消息（环境变量 MSG_RATE_LIMIT，默认 10）
- 超出限流返回错误，不断开连接

### 3.2 连接限流

**位置：** `internal/handler/ws.go` → `ServeWS`

**实现：** 在 `upgrader.Upgrade()` 之前检查当前连接数：

```go
func (h *Hub) ServeWS(w http.ResponseWriter, r *http.Request) {
    if len(h.conns) >= h.maxConns {
        http.Error(w, "connection limit reached", http.StatusServiceUnavailable)
        return
    }
    conn, err := upgrader.Upgrade(w, r, nil)
    // ...
}
```

**变更：** `Hub` 结构体新增 `maxConns int` 字段，`NewHub` 接收该参数。

### 3.3 消息频率限制

**位置：** `internal/middleware/ratelimit.go`（新文件）

**算法：** `golang.org/x/time/rate` 令牌桶

**实现：**

```go
package middleware

import "golang.org/x/time/rate"

type RateLimiter struct {
    limit    rate.Limit
    burst    int
    limiters map[int64]*rate.Limiter // per QQ
    mu       sync.Mutex
}

func NewRateLimiter(msgPerSec int) *RateLimiter {
    return &RateLimiter{
        limit:    rate.Limit(msgPerSec),
        burst:    msgPerSec,
        limiters: make(map[int64]*rate.Limiter),
    }
}

func (rl *RateLimiter) GetLimiter(qq int64) *rate.Limiter {
    rl.mu.Lock()
    defer rl.mu.Unlock()
    lim, ok := rl.limiters[qq]
    if !ok {
        lim = rate.NewLimiter(rl.limit, rl.burst)
        rl.limiters[qq] = lim
    }
    return lim
}

func (rl *RateLimiter) Allow(qq int64) bool {
    return rl.GetLimiter(qq).Allow()
}

func (rl *RateLimiter) Remove(qq int64) {
    rl.mu.Lock()
    defer rl.mu.Unlock()
    delete(rl.limiters, qq)
}
```

**集成到 Hub：**

- `Hub` 新增 `rateLimiter *middleware.RateLimiter` 字段
- `NewHub` 根据 `MSG_RATE_LIMIT` 创建限流器
- `handleConnection` 中认证后，每条消息 dispatch 前检查限流

**dispatch 变更：**

```go
func (h *Hub) dispatch(c *ws.Conn, msg *model.Message) {
    // 登录/注册消息不限流
    if msg.MsgType == model.MsgTypeLogin || msg.MsgType == model.MsgTypeRegister {
        // ... handle normally
        return
    }
    // 已认证用户检查限流
    if c.QQ != 0 && h.rateLimiter != nil && !h.rateLimiter.Allow(c.QQ) {
        h.sendError(c, "rate limit exceeded")
        return
    }
    // ... normal dispatch
}
```

### 3.4 配置

**`internal/config/config.go` 新增：**

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `MSG_RATE_LIMIT` | `10` | 每用户每秒最大消息数 |

### 3.5 错误消息

复用现有 `MsgTypeError = 102`，Content 为错误描述。

### 3.6 清理机制

用户断开连接时，`handleConnection` 退出时调用 `rateLimiter.Remove(qq)` 清理限流器，避免内存泄漏。

### 3.7 验证方式

```bash

# 启动 3 个客户端，第 3 个应被拒绝

# 消息频率测试

# 快速发送 5 条消息，后 2 条应返回 "rate limit exceeded"
```
