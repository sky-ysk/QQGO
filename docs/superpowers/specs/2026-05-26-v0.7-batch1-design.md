## 1. SQLite GREATEST Compatibility Fix

### Problem
`internal/service/chat.go:568` uses `GREATEST(member_cnt - 1, 0)` which is not supported by SQLite. SQLite uses `MAX()` as the scalar equivalent.

### Fix
Replace:
```go
gorm.Expr("GREATEST(member_cnt - 1, 0)")
```
With:
```go
gorm.Expr("MAX(member_cnt - 1, 0)")
```

### Verification
- Existing `TestGroupChat` already exercises `LeaveGroup` — verify it passes without SQLite errors
- Run full test suite to confirm no regressions

## 2. BUG-005: Friend Limit Boundary Test

## 3. BUG-006: Offline Friend Request Persistence

### Current Behavior
Friend requests are stored in `friends` table with `status=0` (pending). When target is offline, the request persists. When target comes online, `/friends` shows pending requests. This flow is implemented but untested end-to-end.

### Test Plan
Create `TestOfflineFriendRequest` in `chat_test.go`:
1. Register user A and user B
2. A sends friend request to B (B is "offline" — no WebSocket connection, just service-level test)
3. Verify `friends` table has a record with `status=pending`, `qq=A, friend_qq=B`
4. Call `GetFriendList(B)` — verify the pending request appears in the "待处理" section
5. B accepts the request — verify status changes to `accepted`
6. Verify both A and B see each other in their friend lists

### Note
This is a service-level test (no WebSocket). The handler layer's `notifyFriendRequest` handles online notification, which is already covered by integration tests. The service-level persistence is what needs testing.
