  - 不要每次测试的时候，都要删除数据库，怎么做？是需要数据库表的可扩展性吗？
  - 可以先做成必须加了好友才能发消息，后续扩展成非好友的话，必须用户存在才能发消息
  - 应该有一个登录选项，每次运行client的时候，如果没有输入用户名和密码，那么应该需要使用/login接口提供登录服务，而不是需要再次退出运行程序才能登录。
  - 加一个功能，能查看当前账号的信息，至少包括uid name
  - 后续加入备注功能
  - 加一个接口，把数据库的
  - /register的时候，应该用户名在前，密码在后
  - 我觉得登录的逻辑需要修改，client运行的时候把目前客户端登录需要的参数删除，先进到界面里，再调用login或者register，参考目前QQ的实现方式。

- bug：
  - 有一个bug，当/to不存在的用户时，没有做逻辑上错误的处理，而是把这条不应该存在的消息存到数据库里了。
  - 用户退出的时候，仍然打印了这个2026/05/21 17:21:54 [client] read error: read tcp [::1]:54379->[::1]:8080: use of closed network connection。这个应该不是bug，而是客户端没有优雅退出导致的吧？如果是这个原因，那么应该设计一种优雅退出的方式。
  - 有个问题，当只使用了name运行了客户端，那么后续注册的时候，它自动忽略了我输入的password username里面的后面这个username，而是把这个参数给我自动变成了nickname，现在的uid和nickname很容易让人搞乱，注册的时候名字就是uid，默认nickname和uid是一样的，只有其他人才能手动修改自己界面显示的自己好友的nickname。


## 2026-05-25 第一次测试-v0.5版本
- bug:
 - /发送消息后，接收方有反应，但是打印顺序有问题，接受到hello但是打印的是“[10002 -> 10001]: nihao:10002] >”，消息被打印到中间去了
 - 当在一个群组对话窗的时候，使用/leavegroup之后，无法退出，使用/to 10001虽然打印了10002和10001之间的对话，但是仍然困在群组的聊天，导致发任何消息都被发到群组里，并且被服务端判定为非群组成员，需要紧急修复，并且自动测试的时候要测这个

- feature:
 - 不知道目前是否持久化了用户的聊天记录（包括个人之间的和群组里面的），如果没有，那么需要规划这个功能
 - 群组也需要加入/prev /next查找历史聊天记录，这个肯定要求保存了本地聊天记录。目前QQ的方式是在固定位置按照QQ号作为文件夹来保存各自的聊天信息和文件之类的。这个是需要做的一个功能，在根目录新建一个DATA文件，在这里按照QQ号来区分各自的聊天记录保存的位置。
 - 另外，无法给非群组成员的群发消息，这点需要优化，因为虽然群组不会接收到非成员的消息，但是服务端还是收到了消息，应该在用户端就避免发送消息，发消息前如果非群组成员，那么无法进入发送窗口。如果发消息的时候检查到非成员，那么自动退出当前群组对话窗口。
